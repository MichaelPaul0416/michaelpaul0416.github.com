索引分类

  1. 存储结构划分
    1. BTree索引
    2. 哈希索引
    3. full-index索引
  2. 应用层次划分
    1. 普通索引：一个索引只包含单个列，一个表可以有多个单列索引
    2. 唯一索引：索引列的值必须唯一，但是允许有空值
    3. 符合索引：一个索引包含多个列
  3. 表记录的排列顺序和索引顺序划分
    1. 聚集索引：表记录的排列顺序和索引的排列顺序一致
    2. 非聚集索引：表记录的排列顺序和索引的排列顺序不一致

磁盘io花费时间

1.寻道时间

磁臂移动到指定磁道所需要的时间，一般在5ms以下

2.旋转延迟

指的是磁盘转速

3.传输时间

从磁盘读出或者将数据写入磁盘的时间，一般在零点几毫秒，相较于前两个时间可以忽略不计

m阶的B-Tree的特点

1.每一个节点最多有m个子节点

2.每一个非叶子节点（除根节点）最少有m/2个子节点

3.如果根节点不是叶子节点，那么它至少有两个子节点

4.有k个子节点的非叶子节点拥有k-1个键[键：主键值]

5.所有的叶子节点都在同一层

比较：B-Tree寻找的过程分为加载磁盘块+内存比较寻找；由于内存寻找的对象本身就是有序的，同时可以利用二分查找法进一步提高效率，所以整个过程中瓶颈在于磁盘操作加载磁盘块；B-Tree相对于AVLTree缩减了节点个数，使每次磁盘IO取到内存的数据都发挥了作用，从而提高效率

B+Tree是在B-Tree上的一种优化；它的所有数据记录节点都是按照键值大小顺序存放在最尾层的叶子节点上，而非叶子节点只存储key的信息
这样对于同一块磁盘块，就能存储更多的key，降低B+Tree的高度

建立索引的原则

最左前缀匹配 mysql会一直向右匹配一直到遇到范围查询[>,<,between,like]就停止匹配 比如where a=1 and b=2 and c>3 and d=4，此时建立(a,b,c,d)的联合索引，那么d索引是用不到的

=和in可以乱序 没有前后顺序区分

尽量选择区分度高的列作为索引 公式是count(dictinct column)/count(*) 表示字段不重复的比例，比例越大表示区分度越高，主键的区分度为1，一般需要join的字段都要求是0.1以上

索引列不参与计算 B+Tree中村的都是数据表的字段值，进行检索的时候，需要把所有元素都使用函数计算后才能得到，这样成本太大

尽量扩展索引，不要新建索引 比如表中已经有a的索引，现在要加(a,b)的索引，那么只需要修改原来的索引即可

explain

Id

> 有几个select就有几个id；select查询分为simple简单查询和primary复杂查询；复杂查询由（简单子查询，派生表：from语句中的子查询，union子查询组成）
> id列越大执行的优先级越高，id相同则从上往下执行，id为null最后执行

select_type

> 标识对应行是简单查询还是复杂的查询

table

> 标识explain这一行正在访问哪个表；
> 当from子句中有子查询的时候，table列是格式，标识当前查询依赖id=N的查询
> 当有union的时候，union result的table的列的值是<union 1,2> 1,2标识参与union的select行id

partitions

> 匹配的分区

type

> 标识关联类型或者访问类型，最优到最差的类型分别是
> system > const > eq_ref > ref > range > index > all
> NULL:mysql能够在优化节点分解查询语句，在执行阶段就不用再访问表或者索引
> const,system：查询时对部分进行优化并且加将其转换成常量。primary key或者unique key的所有列与常数比较的时候，最多只有一个匹配行
> eq_ref：primary_key，unique_key索引的所有部分被链接使用，最多只会返回一条符合条件的记录。除const之外最好的链接类型
> ref：与eq_ref相比，不使用唯一索引，使用普通索引或者唯一索性的部分前缀，索引要和某个值比较，可能会找到多个符合条件的行
> range：通常出现在in,between,>,<,>=等范围比较的操作中
> index：使用扫描全表索引，比ALL快一些。index是从索引中提取的，all是从读盘中读取
> all：全表扫描，mysql需要从头到尾查找所需要的行。

possible_keys

> 显示查询可能使用哪些索引来查找。
> 可能会出现possible_keys有列，但是key显示NULL的情况，这种情况说明mysql觉得这张表中的数据不多，索引对查询的帮助不大
> 如果这一列是null，那么就没有相关的索引

key

> mysql实际采用哪个索引来优化对表的访问

key_len

> mysql在索引里使用的字节数，可以算出具体使用索引中的哪些列

ref

> 在key列记录的索引中，表查找值所用到的列或者常量

rows

> mysql估计需要读取并且检测的行数，并不是最终结果集里面的行数

filtered

Extra

> using index:查询的列被索引覆盖，并且where筛选条件是索引的最左侧索引。这个出现的话一般是使用了覆盖索引
> using where:查询的列未被索引覆盖，where筛选条件不是覆盖索引的最左侧索引
> using where using index:查询的列没有被索引覆盖，并且where筛选条件是索引列之一，但不是索引的最左前缀列，无法直接通过索引来查找符合条件的数据
> null:查询的列没有被索引覆盖，并且where筛选条件是索引的最左侧索引，意味着用到了索引，但是部分字段没有被索引覆盖，需要通过回表实现，不是纯粹的用到了索引，也不是完全没用到索引
> using index condition:与using where类似，查询的列不完全被索引覆盖，where条件中是一个最左前缀的范围
> using temporary:需要创建一张临时表来处理查询，出现这样的情况一般是需要优化的
> using filesort:对结果使用一个外部索引排序，而不是按照索引次数从表里面读取行。此时mysql会根据链接类型浏览所有符合条件的记录，并且保存排序关键字和行指针，然后排序关键字并按顺序检索行信息
